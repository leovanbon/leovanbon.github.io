## The validator inside the gate

I loaded the dumped bin to IDA. *Luckily, thing wasn't so tangled*

First, jumps to `loc_2AA`

![](attachment/2f857040b1078b8c15985ae51b74f6d1.png)
![](attachment/18f84ac45484ed55fc4568bab1517bb3.png)

There, it calls `sub_275`. This walks the input, uppercases the characters and returns the length. 

![](attachment/7c1932889a21d72012f998d9b128d8a2.png)

![](attachment/f42941e779350fe4c5138cc811897060.png)

So here `flag_len = 40`

Next is `loc_6D`.
Here it calls `sub_2D1`, this thing does some checking, and switches back to 32-bit mode.
The main validation logic lives here

![](attachment/d40bfa3fd89117158a51df37756b42fc.png)

`r11` holds the uppercased input
`r9` holds the original input

The while loops calling `sub_124` checking through `r11`.
The sequence of `sub_F` calculates and checks the hash of flag parts.

### Finishing algos

![](attachment/4db4066b820aa076220645d7895c79de.png)

```
v4 = Ax^3 + Bx^2 + Cx + D
A,B,C,D are derived from a2

v3 = rounded(v4)
```

But where is that second argument? Tracking back, it was poped from the stack.

![](attachment/77dfb709d6f085bbfb3db6b343a1245f.png)

We jumped from `0x6D`, so the top of the stack is `0x72`. There is the polynomial coefficients table

![](attachment/22dff2451d6aacb69c343def31d2ca83.png)

From here, we can dump out the table, replicate `sub_124` and get the uppercased flag
`KALMAR{M15S_TH3_S1GN5_4ND_3NTER_TH3_M4Z3}`

Then, we can replicate `sub_F` and brute force to get the correct casing of the characters.
